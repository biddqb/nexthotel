## Problem Statement

Vietnamese hotels (10–60 room independent properties) lose 15–25% of revenue to OTAs (Booking.com, Agoda, Traveloka) because they have no usable direct booking channel. Their current stack — OTA listings + a basic Wix/Wordpress site + manual handling via Zalo/phone — is fragmented, leaks revenue, and gives them no real-time inventory or guest data ownership.

The proposal: a cloud product that combines (1) direct booking engine, (2) cloud-hosted PMS (port of nextHotel desktop), (3) static-page CMS for SEO/info pages, (4) per-customer deployments with multi-branch support, packaged as a framework adaptable beyond hospitality.

## Demand Evidence

**Honest assessment: weak.** The founder's current stance is "build first, validate during/after." No specific hotel has explicitly asked for the cloud product. No one has offered to pay. No customer of nextHotel desktop has said "I'd switch to cloud if you built it."

What is true:
- The OTA-fee pain is real and quantifiable (~20% of revenue on a $30K/month hotel = $6K/month)
- The founder has nextHotel desktop deployed/deploying at a real hotel — direct access to one user
- The Vietnamese small-hotel segment is underserved by existing tools (Cloudbeds, SiteMinder, ezCloud are either too expensive, English-first, or designed for 100+ room hotels)

What is not validated:
- Whether hotels prefer an integrated all-in-one over best-of-breed separates
- Whether they will switch from their current OTA + Wix + manual workflow without a hard cost trigger
- Whether the price point ($30–$80/month/hotel?) clears their mental threshold
- Whether a generic CMS inside the PMS is a real selling point (or whether they'll keep using Wix/agencies)

**This is the dominant risk in the plan.**

## Status Quo

Vietnamese small hotels currently use:
- **OTAs (Booking.com, Agoda, Traveloka)** for visibility — pay 15–25% commission per booking
- **A basic website** (Wix, Wordpress, sometimes a Facebook page acting as the website) — no real-time availability, often outdated
- **Manual handling** for direct inquiries — guest messages on Zalo/phone, hotel staff types it into Excel or nothing

Cost of this status quo:
- 15–25% of revenue gone to OTAs
- Time spent manually answering inquiries and copying bookings between systems
- No guest data ownership (OTAs own the customer relationship)
- No way to do follow-up marketing, loyalty, or repeat-stay offers

It works. It's expensive. Most hotels live with it because the alternative tools (Cloudbeds, SiteMinder) are either too pricey or English-first or both.

## Target User & Narrowest Wedge

**Target:** Owner of a 10–60 room independent hotel in Vietnam, currently using OTAs + a basic website, who personally signs the OTA invoices and feels the commission pain monthly.

**Wedge (deferred):** The founder rejected starting with a narrow wedge (booking widget only). Plan is to ship the full integrated product. **This is acknowledged as the highest-risk decision in the plan.** If the product fails to find paying customers within 6 months of launch, the team should reconsider whether the wedge needed to be narrower.

## Constraints

- Vietnamese-first UI (English and other languages deferred to post-launch)
- Per-customer deployments: each customer chain gets its own database and instance
- Multi-branch support inside a single customer (one chain → N hotels)
- Money in integer VND, dates as YYYY-MM-DD strings (carry forward from desktop)
- Must reuse as much nextHotel desktop code as possible to minimize rewrite cost
- Built by a small team (1–3 engineers including the founder)
- Targeting <6 months from start to first paying customer

## Premises

User-accepted premises (the plan depends on these being true):
1. **Demand for integrated all-in-one** over best-of-breed separates — among Vietnamese small hotels.
2. **Switching is possible without referenceable customers** in the first 3–6 months.
3. **Framework abstraction** can stay sharp without becoming generic mush.
4. **Execution: 4 product surfaces in 6 months** at trust-grade quality (booking + PMS + CMS + multi-branch + per-customer deployment).
5. **Moat** over Cloudbeds / SiteMinder / ezCloud is a single-sentence story (Vietnamese-first + integrated + per-customer data isolation).
6. **CMS inside PMS** is a real value driver, not a "nice to have" hotels skip in favor of Wix/agencies.

User flagged for de-risking: per the office hours conversation, the founder did not explicitly call out which premises are most fragile, but the absence of demand evidence (Q1 = "It's my hypothesis") makes premise 1 and 2 the highest-risk by default.

## Approaches Considered

### Approach A: Cloud nextHotel + Bolt-Ons (CHOSEN)
Port existing Rust/Axum + SQLite app to cloud. Migrate SQLite → PostgreSQL. Add multi-branch support (one tenant → N hotels). Add booking widget as embeddable JS. Add a simple page builder CMS. Deploy each customer to their own Docker/VM with their own database. Effort: L (3–4 months). Risk: Med.

### Approach B: Booking Widget First
Ship just a standalone embeddable booking JS widget. Hotels paste a `<script>` tag on their existing site. Real-time availability + payment. Defer CMS, cloud PMS, multi-branch to v2. Effort: M (6–8 weeks). Risk: Low–Med. **Validates demand before full build.**

### Approach C: Headless Hospitality Core + Plugin Frontends
Architect domain logic (rooms, bookings, guests, rates, payments, branches) as a clean library + JSON API. Frontends (cloud PMS, booking widget, CMS, mobile) plug in independently. Multi-vertical (hotels → spas → restaurants) via configuration. Effort: XL (6+ months). Risk: High. **Highest abstraction risk.**

## Recommended Approach

**Chosen: Approach A — Cloud nextHotel + Bolt-Ons.**

### Architecture sketch

```
Per-customer deployment (one Docker stack per chain):
┌──────────────────────────────────────────────────┐
│  nexthotel-cloud (single binary, like desktop)    │
│  ├─ /app/*    cloud PMS web UI (React)            │
│  ├─ /book/*   public booking widget (React)       │
│  ├─ /pages/*  CMS-rendered marketing/info pages   │
│  ├─ /api/*    Axum API (existing + new)           │
│  └─ PostgreSQL (own DB, own row for tenant)       │
└──────────────────────────────────────────────────┘
       ↑
       │ (managed hosting per customer chain — Hostinger, DigitalOcean, etc.)
```

### Key technical changes from desktop

1. **SQLite → PostgreSQL.** Most rusqlite calls translate, but transactions, full-text search, and migrations need updating. Budget: 2–3 weeks.
2. **Multi-branch model.** Add `branch_id` to all relevant tables. UI gets a branch picker. Cross-branch reports added. Budget: 2–3 weeks.
3. **Booking widget.** Public-facing React app at `/book/<branch-slug>`. Real-time availability via WebSocket or polling. Stripe + VNPay/MoMo for Vietnamese payment rails. Budget: 4–5 weeks.
4. **CMS module.** Pages have title, slug, body (Markdown or block editor), category, SEO meta. Renders at `/pages/<slug>`. Budget: 2–3 weeks.
5. **Auth changes.** Desktop = local PINs. Cloud = email + password (or PIN with rate limiting + MFA). Public booking is anonymous. Budget: 1–2 weeks.
6. **Deployment automation.** Docker image + provisioning script per customer. Backup automation. Update flow (already have self-updater foundation). Budget: 2–3 weeks.

Total: ~3–4 months for one engineer at full speed. With another engineer, ~2–3 months.

### Framework piece (deferred)

The "adaptable to other verticals" goal is achieved by **clean code organization, not architectural abstraction.** Keep all hospitality-specific logic (rooms, rate plans, housekeeping) in well-named modules. When the first non-hotel customer asks (spa, clinic, restaurant), fork the codebase and adapt. Don't pre-abstract. **Repeat: don't pre-abstract.**

## Open Questions

1. **Pricing model.** Per booking, flat monthly per branch, or revenue share? Founder hasn't decided.
2. **Payment rails.** Stripe doesn't work in Vietnam. VNPay, MoMo, OnePay are the options. Each has integration cost.
3. **Hosting.** Self-host on customer's existing infra, or managed hosting on Hostinger/DigitalOcean? Affects pricing + support load.
4. **OTA channel manager.** Will the cloud product also push availability to Booking.com / Agoda / Traveloka? If yes, that's a separate large work item (channel manager APIs are notoriously complex). If no, hotels still need to manually update OTAs.
5. **Data migration from nextHotel desktop.** Existing nextHotel desktop users — how do they migrate to cloud? Is there a one-click import tool?
6. **Domain / SEO.** Each customer has their own domain (myhotel.com)? Subdomain (myhotel.nexthotel.cloud)? Both?
7. **Who is the design partner?** The plan accepts the risk of building without explicit customer pre-commitment. But the founder has at least one nextHotel desktop user. **Strong recommendation: get that customer involved in cloud design from week 1, even informally. Free beta, weekly check-ins, watch them use the booking widget on their site.**

## Success Criteria

By month 6 (end of initial build):
- Booking engine live on at least 1 hotel's domain, taking real bookings
- Cloud PMS deployed for at least 1 customer (chain owner can log in from phone/laptop)
- CMS used by that customer for at least their homepage and 1 info page
- 1 paying customer at $30+/month/branch (or revenue share equivalent)
- Decision point at month 6: (a) double down with capital + sales, (b) pivot wedge if demand isn't materializing, (c) sunset cloud and stay desktop-only

## Distribution Plan

- **Build artifact:** Docker image, hosted on GitHub Container Registry (ghcr.io/biddqb/nexthotel-cloud)
- **CI/CD:** GitHub Actions builds image on tag push, runs tests, publishes to ghcr.io
- **Customer onboarding:** managed deployment per customer (you provision a VM, deploy the image, hand them a URL). Self-host option for technical customers — they pull the image themselves.
- **Updates:** Same self-update pattern as desktop, but applied to the cloud instance. Or rely on Docker image pull + restart per customer.
- **Public-facing booking widget:** built as a static JS bundle, served from a CDN. Hotels embed via `<script src="https://cdn.nexthotel.cloud/widget.js" data-branch="customer-slug"></script>`.

## Dependencies

- PostgreSQL skills on the team (or decide to keep SQLite — viable up to ~100 concurrent users per instance)
- Vietnamese payment gateway integration (VNPay / MoMo / OnePay) — pick one for v1
- Docker / VM provisioning workflow
- Domain management (each customer's chosen domain pointed at their instance)
- Email-sending service (for booking confirmations) — SendGrid, Postmark, or Mailgun
- Image storage (for hotel photos in CMS) — S3-compatible (DigitalOcean Spaces, Backblaze B2, Cloudflare R2)
