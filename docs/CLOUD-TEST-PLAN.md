## Affected Pages/Routes

### Public (no auth)
- `/book/<branch-slug>` — booking widget landing
- `/book/<branch-slug>/availability?from=&to=` — availability JSON
- `POST /book/<branch-slug>/reserve` — create reservation
- `/pages/<slug>` — CMS-rendered marketing pages (Astro SSR)
- `GET /api/vnpay/callback` — VNPay return URL after payment
- `POST /api/vnpay/webhook` — VNPay server-to-server notification

### Authenticated — Admin/Manager (email + password)
- `/app/login` — email/password form
- `/app/dashboard` — overview
- `/app/branches` — branch picker, switch active branch
- `/app/cms` — page list + editor
- `/app/settings` — hotel info, payment config

### Authenticated — Staff (PIN)
- `/app/staff-login` — PIN form
- `/app/calendar` — same as desktop
- `/app/housekeeping` — same as desktop
- All other PMS pages reused from desktop

## Key Interactions to Verify

### Public booking widget
- Visit `/book/<branch-slug>` from anonymous browser → widget loads with branding
- Pick check-in + check-out dates → availability list updates
- Pick a room → click "Đặt phòng" → redirect to VNPay
- Complete payment in VNPay → return to confirmation page
- Refresh confirmation page → same reservation, no duplicate
- Receive booking confirmation email in Vietnamese

### Branch switching
- Manager has access to 3 branches → branch picker shows all 3
- Switch from Branch A to Branch B → calendar shows Branch B's reservations only
- Reports default to active branch only
- "All branches" toggle in reports → aggregates across branches

### CMS
- Manager creates page "Giới thiệu" → saves → page renders at `/pages/gioi-thieu`
- Markdown formatting (headings, links, images) renders correctly
- Page slug uniqueness enforced within branch
- Draft vs published — drafts not visible publicly

### Auth
- Manager logs in with email + password → redirected to dashboard
- Staff logs in with username + PIN → redirected to calendar (PMS-only access)
- Manager session persists 30 days, staff session persists 30 days
- Logout clears session
- MFA enrollment and challenge flow (if enabled)

## Edge Cases

### Concurrent booking race ⭐ CRITICAL
- Two browser tabs, both reserve the same room same dates → exactly one succeeds
- Race window: between availability check and INSERT
- Acceptance: second request returns 409 Conflict with message "Phòng vừa được đặt, vui lòng chọn phòng khác"

### Payment-reservation atomicity ⭐ CRITICAL
- VNPay charges card successfully → backend crashes before INSERT → user has paid but no reservation
- Acceptance: Idempotency key per booking attempt, reconciliation worker checks all VNPay payments against reservations daily, alerts if mismatch

### VNPay edge cases
- VNPay returns timeout (>30s) → user-facing message in Vietnamese, no charge made
- VNPay returns "decline" → reservation not created, friendly message shown
- VNPay webhook arrives before browser callback → reservation already exists, webhook is no-op
- VNPay webhook with invalid HMAC signature → 400 rejected, logged
- Duplicate webhook (VNPay retries) → idempotent

### Cross-branch isolation ⭐ CRITICAL
- Manager A logs in → cannot see Manager B's branch data even with crafted URL parameters
- Test: hand-craft `/api/reservations?branch_id=999` → blocked by RLS
- Test: SQL query without branch_id filter → RLS returns zero rows

### Date / availability
- Check-out date <= check-in date → 400 validation error
- Date in past → 400 validation error
- Date >2 years in future → 400 validation error (sanity check)
- Cross-month booking → spans correctly
- Rate plan boundary (last day of "summer rate") → correct rate per night

### CMS
- Slug with special chars or duplicates → friendly validation error
- Body with malicious markdown (XSS attempts) → sanitized
- Image upload >10MB → rejected
- Page deleted while being edited (concurrent admin) → 409 conflict

### Empty/zero/boundary states
- Branch with zero rooms → calendar shows empty state, booking widget says "no rooms"
- Hotel with zero CMS pages → /pages root redirects somewhere sensible
- Year-end / leap-year / DST edge cases on dates

### Network failures (booking widget)
- Slow API response (>5s) → loading indicator, doesn't double-submit
- API returns 500 → friendly Vietnamese error, retry button
- VNPay popup closed without paying → graceful return to widget
- Browser back button after payment → idempotent, no duplicate reservation

### Multi-process restart
- Astro SSR restarts mid-request → Caddy retries against API only, request completes
- Axum restarts mid-booking → in-flight VNPay webhook arrives → idempotency key prevents double-write

## Critical Paths (E2E required)

These flows MUST have E2E tests in Playwright before any production launch:

1. **Complete public booking** — visit widget → select dates → select room → VNPay sandbox → confirmation page → email received
2. **Manager onboarding** — login → first-run wizard → add branch → invite staff → first reservation created via PMS
3. **Cross-branch isolation** — login as Branch A manager → attempt to read Branch B URLs → all blocked
4. **Payment failure recovery** — start booking → payment declined → user retries → new attempt succeeds → no orphan records

## Test infrastructure

**Frameworks to add:**
- Rust: `cargo test` (already), add `sqlx::test` for Postgres integration tests with auto-rollback
- Postgres in CI: `services: postgres:16` in GitHub Actions, or Testcontainers for local
- Frontend: `vitest` + `@testing-library/react`, jsdom env
- E2E: `Playwright` with VNPay sandbox, runs against staging instance in CI

**Coverage targets:**
- core/: 95%+ (pure domain, no excuse for gaps)
- cloud/repo_postgres.rs: 90%+ (every query, every RLS path)
- cloud/public_widget.rs: 100% (public-facing, all error paths covered)
- cloud/vnpay.rs: 90%+ (every API response variant, every webhook case)
- cloud/cms.rs: 80%+
- Frontend pages: 70%+ for critical paths only (don't over-test view code)
- E2E: 4 critical paths above (always green, blocks deploy)

**Regression tests required:**
- Reservation overlap detection — port from desktop, verify still works in PG
- Rate plan resolution — port from desktop, verify with multi-branch context
- Night audit calculations — port from desktop, verify totals across branches
