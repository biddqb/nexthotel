# Design: nextHotel — Offline-Capable Desktop PMS for a 12-Room Hotel

## Problem Statement

The owner-operator of a 12-room independent hotel in Vietnam currently runs the entire property — bookings, guests, payments, reporting — on Microsoft Excel. A previous paid PMS (property management system) was abandoned because the monthly subscription cost was not justified for a 12-room property. Excel is error-prone (no data integrity, no calendar view, no concurrent safety), painful to generate reports from, and does not produce professional invoices or receipts.

The user wants a **local-first desktop application** that replaces the Excel workflow, runs on their laptop with no cloud backend (zero recurring infrastructure cost), and is usable in Vietnamese.

## Demand Evidence

- **User is user #1** — owns and operates the hotel, feels the pain daily.
- **Paid software was tried and abandoned** — strong negative signal on existing SaaS PMS pricing for small properties.
- **Excel is the shadow product** — confirms the workflow is real and currently painful.
- **No external demand evidence yet** — user has not interviewed other hotel owners. This is a known gap and is deliberately deferred (see "The Assignment" below).

## Status Quo

- Paid PMS software: churned out due to cost.
- Excel: tracks bookings, income, and everything else. No calendar grid, no validation, single file, high risk of data loss/corruption. Manual report generation.

## Target User & Narrowest Wedge

- **Primary user (v1):** the hotel owner themselves. Solo operator. Vietnamese language.
- **Secondary user (v2, deferred):** other independent Vietnamese hotel owners running 10–30 rooms on Excel or abandoned SaaS. Massive long-tail segment.
- **Narrowest wedge:** one-for-one Excel replacement with a real calendar UI, data integrity, and printable invoices. Ship in a few weekends. Use it for the user's own hotel. Decide later whether to productize.

## Constraints

- **Platform:** Tauri v2, desktop only (Windows primary, macOS nice-to-have).
- **No cloud backend.** No monthly infrastructure cost.
- **Vietnamese UI.** First-class localization, not an afterthought.
- **Solo developer (the user).** Building with Claude Code assistance.
- **Budget: weekends, not months.**

## Premises (all agreed)

1. MVP goal is **replace the user's Excel workflow** — not compete with Cloudbeds/Little Hotelier/Hotelogix. Not multi-property. Not multi-user.
2. **Local-first: Tauri v2 + SQLite on the user's machine.** Zero cloud cost.
3. **Manual data entry for v1.** No OTA sync (Booking.com, Airbnb, Agoda), no channel manager, no payment gateway.
4. **Single-user, single-device.** No sync, no multi-staff accounts.
5. **Vietnamese UI is a v1 requirement**, not deferred.

## Approaches Considered

### Approach A: Minimal Viable — "Just Replace My Excel"
Tauri v2 + React/Svelte + SQLite. Single window, single user. Covers MVP feature set. No architecture concessions for future multi-tenancy. Ships in 2–4 weekends. Low risk. Rewrite the data layer later if productizing.

### Approach B: Ideal Architecture — "Build v1 With v2 in Mind"
Same feature set as A, but data model designed multi-tenant from day one (every row has `hotel_id`), clean repository layer that can later swap SQLite for libSQL embedded replicas, i18n scaffolding ready for more locales. 2x the v1 work for hypothetical future value. Medium risk. Classic YAGNI trap if productization never happens.

### Approach C: Lateral — "Polished Google Sheet + Tauri Dashboard"
Google Sheets remains the data store; Tauri reads via API and renders calendar/reports/invoices. Fastest validation. Violates the no-cloud premise. Not a productizable path. Rejected.

## Recommended Approach

**Approach A**, with one small concession: every domain table includes a `hotel_id` column from day one (hardcoded to `1` in v1). Costs nothing now, saves a migration if productization happens.

### Feature Set (v1 MVP)

1. **Reservation calendar** — month grid view, rooms on Y axis, days on X axis. Click-drag to create a booking. Visual status (confirmed, checked-in, checked-out, cancelled).
2. **Guest records** — name, phone, ID/passport number, nationality, notes. One guest can have many reservations.
3. **Reservation CRUD** — create, edit, cancel. Fields: room, check-in, check-out, guest, rate, deposit, payment status.
4. **Check-in / check-out** — one-click state transitions from the calendar.
5. **Daily income log** — per-day revenue, broken down by room and payment method (cash, bank transfer, card-via-POS).
6. **Reports** — occupancy rate (daily/monthly), revenue (daily/monthly), top-room / repeat-guest lists.
7. **Printable invoices and receipts** — Vietnamese, with hotel branding, tax-ID placeholder, VND formatting.
8. **Local backup** — one-click export to a timestamped SQLite file. User can copy to USB/cloud drive themselves.
9. **Vietnamese UI** — all strings through an i18n layer (even if only one locale exists in v1).

### Tech Stack

- **Shell:** Tauri v2 (Rust backend, WebView frontend).
- **Frontend:** React + TypeScript + Vite (or Svelte — user's choice). Tailwind for styling.
- **Database:** SQLite via `tauri-plugin-sql`. File stored in Tauri's app data directory.
- **State:** TanStack Query for reads, plain Rust commands for writes.
- **i18n:** `react-i18next` (or equivalent) with Vietnamese as default locale.
- **Printing:** HTML → print dialog via Tauri's window API, or generate PDF via a library like `printpdf` in Rust.
- **Packaging:** Tauri bundler → `.msi` (Windows) and `.dmg` (macOS).

### Data Model

All money stored as `INTEGER` VND (no decimals). Dates stored as `TEXT` in `YYYY-MM-DD` format (not timestamps). Every table has `hotel_id` (hardcoded to `1` in v1, multi-tenant seam).

```
hotels (id, name, address, tax_id, phone, ...)                      -- always 1 row in v1
rooms (id, hotel_id, room_number, type, base_rate INT, ...)
rate_plans (id, hotel_id, name, starts_on TEXT, ends_on TEXT,       -- seasonal pricing
            applies_to_room_type, rate INT, priority, ...)
guests (id, hotel_id, name, phone, id_number, nationality, notes, ...)
reservations (id, hotel_id, room_id, guest_id,
              check_in TEXT, check_out TEXT,
              status TEXT CHECK(status IN
                 ('confirmed','checked_in','checked_out','cancelled','no_show')),
              payment_status TEXT CHECK(payment_status IN
                 ('unpaid','deposit_paid','paid','refunded')),
              rate INT NOT NULL,          -- snapshot at booking, never changes
              created_at TEXT, ...)
payments (id, hotel_id, reservation_id, amount INT,
          method TEXT CHECK(method IN ('cash','bank','card','other')),
          paid_at TEXT, notes, ...)
audit_log (id, hotel_id, entity, entity_id, action, diff, created_at)
```

### Reservation state machine

```
                    create
                      ↓
                  confirmed ─────cancel────→ cancelled
                      │                          │
                  check_in               (terminal)
                      ↓
                 checked_in ─────no_show────→ no_show (terminal)
                      │
                  check_out
                      ↓
                checked_out ──(terminal)──
```

### Required SQLite PRAGMAs (on every connection open)

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;        -- OFF by default in SQLite, critical footgun
PRAGMA synchronous = NORMAL;
PRAGMA busy_timeout = 5000;
```

### Indexes (required)

```sql
CREATE INDEX idx_reservations_room_dates ON reservations(room_id, check_in, check_out);
CREATE INDEX idx_reservations_status     ON reservations(status) WHERE status != 'cancelled';
CREATE INDEX idx_payments_reservation    ON payments(reservation_id);
CREATE INDEX idx_rate_plans_dates        ON rate_plans(starts_on, ends_on);
```

### Double-booking prevention (CRITICAL)

Both layers required:

1. **App layer (inside transaction):** before INSERT/UPDATE on reservations, run:
   ```sql
   SELECT 1 FROM reservations
   WHERE room_id = ?1 AND status NOT IN ('cancelled','no_show')
     AND id != ?4          -- ignore self on update
     AND NOT (check_out <= ?2 OR check_in >= ?3)
   LIMIT 1
   ```
   If a row is returned, reject with a domain error (show user "Room X is already booked from A to B").
2. **DB layer (safety net):** trigger that raises on overlap insert/update. Belt and suspenders.

### Currency formatting

- Storage: `INTEGER` đồng (VND has no decimals).
- Display: `new Intl.NumberFormat('vi-VN', { style: 'currency', currency: 'VND' }).format(n)` → `1.500.000 ₫`.
- Never `REAL` / `FLOAT`.

### Date handling

- Stored as `TEXT YYYY-MM-DD` (lexical sort = chronological).
- No timestamps for check-in/check-out (calendar dates, not moments in time).
- Timezone: Vietnam is ICT (UTC+7, no DST). All display in local time.
- Use `date-fns` + `date-fns-tz`. Not moment.

### Backup strategy

- **First-run setup:** user picks data folder. If OneDrive/Google Drive/Dropbox detected, offer as default (free off-site backup).
- **Rolling backup:** on every app close, `VACUUM INTO '{data_dir}/backups/nextHotel-{timestamp}.sqlite'`. Keep last 30 days, delete older.
- **Never** fs-copy an open SQLite file — `VACUUM INTO` is the only safe method.

### i18n convention

- `react-i18next` with Vietnamese (`vi`) as default + only locale for v1.
- Flat keys by domain: `reservation.create.title`, `report.occupancy.header`, `common.actions.save`.
- All strings through `t()` from day one, even though only one locale exists. v2 English is a JSON file, not a refactor.

## Test Coverage Plan (added by /plan-eng-review)

100% of listed paths must have tests at v1 ship. CRITICAL paths block ship.

### Reservation domain (highest risk)
- `reservations::create` — happy path, overlapping dates rejected, adjacent stays allowed, check_out <= check_in rejected, cancelled reservation frees dates, rate snapshotted from rate_plan.
- `reservations::cancel` — confirmed → cancelled frees room, cannot cancel checked_in, refund payment recorded.
- `reservations::check_in` / `check_out` — state transitions enforced, audit row written.

### Calendar UI
- Renders 12 rooms × month grid with empty state.
- [E2E] Drag-to-book covers correct date range.
- Booking crossing month boundary renders correctly in both months.
- Month navigation preserves data (no flicker).

### Money
- `formatVND(1500000)` → `"1.500.000 ₫"`.
- `formatVND(0)` → `"0 ₫"`.
- `formatVND(-500000)` → `"-500.000 ₫"`.
- `payments::record` — full → `paid`, partial → `deposit_paid`, over → still `paid` + flag for refund.

### Reports
- `occupancy(from, to)` math = nights_sold / (rooms × nights_in_period).
- Reservation overlapping [from, to] only counts nights in range.
- Empty period returns zeros, not division-by-zero.
- `revenue(from, to)` matches sum(payments.amount) for period.

### Backup / migration
- [CRITICAL] `VACUUM INTO` produces a file that opens in fresh SQLite with all rows intact.
- Backup during concurrent read doesn't corrupt.
- Rolling retention deletes files older than 30 days.
- Up migration on empty DB creates full schema with all indexes + PRAGMAs.

### Invoice / print
- Multi-night stay renders per-night breakdown.
- All strings are Vietnamese (no English leaks).
- [E2E] Print preview matches final printed output.

## Failure Modes

| Path | Production failure | Mitigation |
|------|-------------------|------------|
| Double-booking | Two guests arrive for Room 3 same night | App-level check in transaction + DB trigger |
| Laptop dies | All hotel data lost | Auto-backup to cloud-synced folder |
| Clock/timezone skew | Date rolls over, off-by-one bookings | YYYY-MM-DD strings only, no timestamps |
| Concurrent writes (two windows) | SQLite lock error | WAL mode + `busy_timeout = 5000` |
| Payment recorded twice | Revenue inflated | Unique constraint `(reservation_id, paid_at, amount)` + test |
| Delete guest with reservations | Orphan rows, crash on report | FK `ON DELETE RESTRICT` + `foreign_keys = ON` |
| Backup during active transaction | Corrupt backup | `VACUUM INTO`, never fs copy |

## Parallelization Strategy

- **Lane A (sequential, blocks everything):** Tauri scaffold → schema → migrations → PRAGMAs → base Rust commands.
- **Lane B (parallel with C after A):** i18n setup, Vietnamese strings, base layout + navigation.
- **Lane C (parallel with B after A):** Invoice/receipt HTML template + print plumbing.
- **Lane D (sequential after A):** Calendar grid — the risky UX, keep focused.
- Then: reports → backup → polish.

## NOT in scope (explicit)

OTA sync, channel manager, payment gateway, multi-user accounts, multi-property, housekeeping module, staff schedules, maintenance tickets, email/SMS notifications, guest-facing booking portal, auto-update infrastructure (deferred to v2), English UI (v2), mobile app.
