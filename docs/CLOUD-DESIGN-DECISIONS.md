## Pass 1 — Information Architecture

### Public booking widget (`/book/<slug>`)

Single page, progressive disclosure. No full reloads between steps.

```
┌────────────────────────────────────────────────────────────┐
│  [Hotel logo]  Khách Sạn ABC                               │  ← brand bar
│  ★★★★ · Đà Nẵng · Tiếng Việt | English                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  [hero photo, full bleed, ~280px tall]                     │  ← visual anchor
│  Khách Sạn ABC                                             │
│  Phòng tiêu chuẩn từ 500.000₫ / đêm                        │
│                                                            │
├────────────────────────────────────────────────────────────┤
│  Đặt phòng                                                 │  ← primary action zone
│  ┌─Ngày nhận─────┐  ┌─Ngày trả──────┐  ┌─Khách─┐          │
│  │ Hôm nay  ▾    │  │ Mai      ▾    │  │ 2  ▾  │  Tìm →   │
│  └───────────────┘  └───────────────┘  └───────┘          │
├────────────────────────────────────────────────────────────┤
│  Phòng còn trống (3)                                       │  ← decision zone
│  ┌──────────────────────────────────────┐                  │
│  │ [photo]  Phòng đôi tiêu chuẩn       │                  │
│  │          1 giường đôi · 2 khách      │                  │
│  │          [wifi] [tv] [ac] [bath]     │                  │
│  │          600.000₫/đêm  ·  Chọn →     │                  │
│  └──────────────────────────────────────┘                  │
│  ... (more rooms) ...                                      │
├────────────────────────────────────────────────────────────┤
│  Thông tin về khách sạn                                    │  ← supporting content
│  Vị trí · Tiện nghi · Chính sách · Liên hệ                 │
└────────────────────────────────────────────────────────────┘
```

**Visual hierarchy (what user sees first → last):**
1. Brand bar (1-2 sec recognition: "is this the right hotel?")
2. Hero photo + name (visceral: "does this look like a place I want to stay?")
3. Date picker (primary action: "let me check my dates")
4. Room cards (decision: "which one fits my needs?")
5. Supporting info (verification: "is this real? what about location/policies?")

**Mobile (≤640px):** same vertical order, single column. Date picker stacks vertically. Room cards full-width. Bottom-fixed "Tiếp tục" bar after a room is selected.

### CMS pages (`/pages/<slug>`)

Default templates that ship with the product. Hotels can edit content; structure is opinionated:

- **Trang chủ (homepage)** — hero photo + brand, "Đặt phòng ngay" CTA → widget, intro paragraph, "Phòng và giá", "Tiện nghi", "Vị trí", footer
- **Giới thiệu (about)** — story, photos, awards/recognition (optional)
- **Phòng và giá (rooms & rates)** — room types with photos, base prices, "Đặt phòng" CTAs
- **Liên hệ (contact)** — address with map, phone, email, WhatsApp/Zalo links

All templates use the same brand bar + footer. Hotels customize content via CMS, layout stays consistent.

### Cloud PMS (`/app/*`)

Mostly identical to desktop. Adds:

- **Branch picker** in the top-left, replacing "nextHotel" wordmark when user has >1 branch:
  ```
  ┌─Khách Sạn ABC ▾─┐  Lịch đặt phòng ...
  │  Chi nhánh:    │
  │  ✓ Đà Nẵng     │
  │   Hà Nội       │
  │   Hội An       │
  │  ─────────────── │
  │  Tất cả chi nhánh│
  └────────────────┘
  ```
- **Reports cross-branch toggle** — when on a branch-specific report, a "Xem tất cả chi nhánh" toggle aggregates data.

### Login flows

Two clearly distinct entry points, no ambiguity:

- `/app/login` — manager/owner: large email field, password field, "Đăng nhập" button, "Quên mật khẩu?" link, MFA prompt if enabled
- `/app/staff-login` — front-desk staff: username + 4-12 digit PIN (large numeric keypad UI on mobile)

Cross-link at bottom of each: "Bạn là nhân viên lễ tân?" / "Bạn là quản lý?"

| CMS page | none (SSR) | "Trang đang được cập nhật" | 404 page with link back to homepage | rendered | draft mode banner if logged in |

**Empty states are features rule applied:** every "no rooms" / "no bookings" / "no pages" state has warmth, context, and a primary action.

**Network failure rule:** every fetch shows a retry button + a friendly Vietnamese explanation. No raw "fetch failed" or "TypeError." All errors map to the i18n table.

## Pass 3 — User Journey & Emotional Arc

Two journeys to design for.

### Journey A: First-time customer booking direct

| Step | User does | User feels | Plan supports it |
|---|---|---|---|
| 1. Lands on widget | sees brand + hero | "is this real? trustworthy?" | brand bar with hotel name, ★ rating, location · hero photo of REAL property (no stock photos) |
| 2. Picks dates | uses date picker | "will my dates be available?" | real-time availability check, fast (<500ms) feedback |
| 3. Sees rooms | scrolls, compares | "which one suits me?" | room cards with photo, occupancy, amenities icons, price |
| 4. Picks a room | clicks "Chọn" | "am I committing? can I undo?" | scroll-into-view booking form, "Quay lại" button visible |
| 5. Fills form | types name + phone | "this hotel has my data now" | minimal fields, only ask what's needed, clear privacy note |
| 6. Sees total | reviews summary | "are there hidden fees?" | itemized total: room × nights, "Thuế và phí (đã bao gồm)", grand total. NEVER add surprise fees later. |
| 7. Pays | clicks VNPay button | "is this safe? am I being scammed?" | VNPay logo, "🔒 Thanh toán bảo mật bởi VNPay", supported card brands listed visibly |
| 8. Returns from VNPay | sees confirmation | "did it actually work?" | clear "Đặt phòng thành công" message, reservation ID shown prominently, email-sent indicator, "Lưu/In hoá đơn" CTA |
| 9. Email arrives | opens inbox | "great, I have proof" | branded email with: hotel name, dates, room, total, contact info, cancellation policy link |

**Emotional arc:** uncertainty → curiosity → consideration → commitment → relief. Each step removes uncertainty.

**Time horizons:**
- 5-second visceral: "this looks like a real Vietnamese hotel, not a sketchy site"
- 5-minute behavioral: "I can find rooms, see prices, book, pay, get confirmation in <5min on my phone"
- 5-year reflective: "I've stayed at this hotel 3 times now, I always book direct, I trust the brand"

### Journey B: Hotel manager checking morning bookings

| Step | User does | User feels | Plan supports it |
|---|---|---|---|
| 1. Opens laptop, navigates | bookmarked URL | "let's see what came in overnight" | fast load, no auth re-prompt within 30-day session |
| 2. Sees dashboard | quick scan | "how many arrivals today?" | "Khách nhận phòng hôm nay" card prominent, count + names |
| 3. Switches branch (if multi) | clicks branch picker | "show me Hà Nội branch" | branch picker fast, persists choice |
| 4. Reviews bookings | clicks reservation | "what's the status?" | drawer opens, all info, no extra clicks |
| 5. Closes laptop | satisfied | "ok, I know what's happening today" | quick wins on calm dashboard |

**Calm by default.** PMS interface should not feel busy or stressful. Hotel managers run on tight margins; the tool should reduce cognitive load.

## Pass 4 — AI Slop Risk

Specific anti-patterns to AVOID:

**Booking widget:**
- ❌ Generic 3-column "Why book direct?" feature grid below the fold ("Best Price Guaranteed!" / "Free WiFi!" / "24/7 Support!")
- ❌ Stock photos of smiling foreign tourists at random hotel pools
- ❌ Centered-everything hero with bubbly large border-radius
- ❌ "Welcome to [Hotel Name]!" / "Unlock the best deals!" / "Your perfect stay awaits"
- ❌ Floating decorative blobs / gradient backgrounds
- ❌ Trust badges that look like clip art
- ❌ Emoji as section decoration

**CMS templates:**
- ❌ The hospitality-template Wordpress look (full-bleed hero, 3-column features, Lorem ipsum-grade copy, generic Unsplash photos)
- ❌ Carousel sliders with no narrative purpose
- ❌ "Subscribe to our newsletter" form everyone ignores

**Cloud PMS:**
- ❌ Heavy card grids replacing the calendar grid
- ❌ Decorative shadows, ornamental icons
- ❌ "Activity feed" / gamification elements

**Specific replacements:**

| Generic | Specific |
|---|---|
| "Card with icon" | "Photo of the actual room with amenity icons inline beneath the price" |
| "Hero section" | "1080×600px crop of the hotel facade or signature room, no overlay text on photo, brand text below in 32px Be Vietnam Pro Bold" |
| "Clean modern UI" | "Hairline borders, 4px border-radius, 16/20/24/32px type scale, 8/16/24/32 spacing scale" |
| "Trust badge" | "VNPay logo (provided by VNPay), 'Thanh toán bảo mật' label, list of accepted card brands" |

## Pass 5 — Design System Alignment

**Cloud must feel like nextHotel desktop.** Reuse the same Tailwind config tokens. Differences only where the audience differs:

- **PMS surfaces (`/app/*`)** — identical density and palette to desktop. Manager already learned the desktop product; cloud should feel like "the same app, in the cloud."
- **Public surfaces (`/book/*`, `/pages/*`)** — slightly more breathing room (24/32/48 spacing instead of 16/24), larger type for hero (32-48px headlines), but same accent + same font + same calm palette. Recognizably the same brand.

**New components needed (add to component vocab):**

| Component | Purpose | Where |
|---|---|---|
| `BrandBar` | Hotel logo + name + lang switch | Public widget + CMS |
| `DatePickerRange` | Check-in/out picker, mobile-friendly | Booking widget |
| `RoomCard` | Photo + type + amenities + price + CTA | Booking widget |
| `BookingSummary` | Itemized total | Booking flow |
| `BranchPicker` | Switch active branch | PMS top bar |
| `PaymentTrust` | VNPay logo + security message | Booking flow |
| `EmptyState` | Vietnamese empty-state component with icon + heading + body + CTA | Everywhere |

**Existing components to reuse from desktop:**
- All status badges (StatusBadge.tsx)
- Modal.tsx, RoomTypeSelect.tsx
- Calendar grid (move from desktop to core/ if not done already)
- All form inputs (already styled in index.css)

## Pass 6 — Responsive & Accessibility

### Responsive specs

**Booking widget (most-bookings happen on phones):**

| Viewport | Layout |
|---|---|
| ≤640px (phone) | single column, date picker stacks vertically, room cards full-width, bottom-fixed CTA bar |
| 641-1024px (tablet) | single column, date picker side-by-side, room cards full-width |
| ≥1025px (desktop) | constrained width 960-1100px, date picker side-by-side, room cards single column with rich layout |

**Cloud PMS:**
- Mobile (≤640px): drawer-pattern nav, calendar collapses to "today list" view (no grid), housekeeping is grid-of-cards (already optimized for tablet in desktop)
- Tablet (641-1024px): primary target for housekeeping
- Desktop (≥1025px): primary target for managers, calendar grid, full chrome

**Touch targets:** 44px minimum on all interactive elements (matches Apple/Google guidelines). Date-picker day cells minimum 44×44px on touch devices.

### Accessibility specs

| Requirement | Spec |
|---|---|
| Keyboard nav | tab through all interactive in logical order, focus ring visible (teal 2px outline + 2px offset) |
| Form labels | always visible, never placeholder-as-label |
| Color contrast | body text ≥ 4.5:1 (slate-900 on white = 16:1, way above) |
| Touch targets | ≥ 44×44px |
| Date picker | keyboard accessible (arrow keys), screen-reader announcements |
| Error messages | aria-live="polite" region for form errors, clear association via aria-describedby |
| Visited links | distinct color (slate-600 instead of teal) — required, not optional |
| Skip link | "Bỏ qua đến nội dung chính" at top of every public page for screen readers |
| Image alt text | every photo must have meaningful alt (room photos: "Phòng đôi tiêu chuẩn, giường đôi, view biển") |
| HTML lang | `<html lang="vi">` on all pages |
| Heading order | h1 once per page, h2 for sections, no skipped levels |

## Pass 7 — Decisions

### Resolved (this review)

| # | Decision | Resolution |
|---|---|---|
| 1 | Booking flow shape | **Single-page progressive disclosure.** No full reloads. Date picker → rooms appear → click room → booking form expands inline. |
| 2 | Payment timing | **Per-hotel setting, default deposit 30%.** Each hotel admin can choose: deposit / full / pay-at-hotel. Default is deposit because it converts best. |
| 3 | Account requirement | **Optional account at end of flow.** Booking is anonymous; AFTER successful booking, offer one-click sign-up: "Lưu thông tin để đặt phòng nhanh hơn lần sau?" |
| 4 | Email field | **Required.** Most Vietnamese customers have email. Reliable confirmation channel. Hotels need this for follow-up communications. |

### Deferred (decide before second customer)

| # | Decision | If deferred default |
|---|---|---|
| 5 | Cancellation policy enforcement at booking time | Default: hotel sets free-cancellation window per room type, displayed before payment, enforced on cancel attempts |
| 6 | Image upload for hotels/rooms — file picker or URL paste | Default: file picker with S3-compatible upload (R2/B2). Same backup setup as DBs. |
| 7 | English/multilingual public pages | Default: Vietnamese-only v1, lang switcher in brand bar marked "(Sắp có)". Add English when first customer asks. |
| 8 | Multi-branch picker pattern | Default: dropdown picker in top-left (works to ~10 branches). Switch to sidebar if a customer has more. |
